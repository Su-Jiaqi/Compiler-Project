## 一、实验目的

本次实验在前面词法分析、语法分析、语义分析以及中间代码生成实验的基础上，进一步理解编译器后端中的运行时环境设计。实验的主要目标是根据语义分析阶段得到的符号表信息，为程序中的变量、临时变量和函数等对象设计合理的存储格式，并构建对应的内存映像，为后续目标代码生成或汇编代码生成提供基础。

通过本次实验，可以进一步理解编译器从源程序到目标程序的完整流程。前面的实验主要关注词法、语法和语义层面的分析，而实验七开始关注程序运行时的数据存储方式，例如变量在内存中的地址、偏移量、作用域以及不同类型变量所占用的空间等内容。

## 五、实验原理

运行时环境是程序执行时所依赖的内存组织方式。编译器在完成语义分析后，需要知道程序中有哪些变量、变量类型是什么、变量位于哪个作用域中，以及它们在运行时如何存储。

对于一个简单的类 C 语言子集，可以采用较为简单的内存模型：

```text
全局区 / 符号表区
局部变量区
临时变量区
中间代码区
```

在本实验中，变量主要来自语义分析阶段生成的符号表。每个符号表项可以包含以下信息：

```text
变量名
变量类型
变量作用域
变量地址或偏移量
```

例如，对于如下源程序：

```c
int a;
int b;
a = 3;
b = a + 5;
print b;
```

可以构建类似如下的符号表和内存映像：

```text
Name    Type    Scope    Offset
a       int     global   0
b       int     global   4
```

其中 `a` 和 `b` 都是整型变量，每个变量占用 4 字节，因此变量 `a` 的偏移量为 0，变量 `b` 的偏移量为 4。

如果后续生成目标代码，就可以根据这些偏移量访问变量。例如中间代码：

```text
t1 = a + 5
b = t1
```

可以进一步翻译为某种汇编风格的目标代码。

## 八、运行时内存映像设计

本项目可以采用如下简单的内存映像设计方式：

```text
Offset    Name    Type    Scope
0         a       int     global
4         b       int     global
8         t1      int     temp
12        t2      int     temp
```

其中：

- 普通整型变量占用 4 字节；
- 浮点型变量可以设置为占用 4 字节或 8 字节；
- 临时变量由中间代码生成阶段产生；
- 每个变量按照声明顺序依次分配偏移量；
- 后续目标代码生成时，可以通过偏移量访问变量。

本实验中的内存映像设计仍然是教学版和简化版，主要用于展示编译器后端从符号表到运行时存储分配的基本过程。

## 九、实验分析

本次实验与前面实验之间具有明显的连续关系。实验二生成 Token 流，实验三和实验四完成语法分析相关内容，实验五生成 AST 和符号表，实验六生成中间代码，而实验七则进一步利用符号表和中间代码进行运行时环境设计。

其中，符号表在实验七中具有核心作用。符号表不仅保存变量名和类型，还可以扩展保存变量的作用域、地址偏移量、是否为临时变量等信息。运行时内存映像本质上就是对符号表中各类对象进行存储空间分配的结果。

由于本项目目前主要实现到中间代码生成阶段，因此实验七的重点是说明如何基于已有符号表进一步扩展内存映像。后续如果继续完善，可以在符号表项中增加 `offset` 字段，并在语义分析或中间代码生成结束后统一分配变量地址，最终输出完整的 `memory_map.txt` 文件。

## 十、实验总结

通过本次实验，我进一步理解了编译器后端中运行时环境和内存映像的作用。前面的词法分析、语法分析和语义分析主要解决的是源程序是否符合语言规则的问题，而运行时环境设计则进一步关注程序在执行时如何组织和访问数据。

本实验基于前面生成的符号表和中间代码，设计了简单的变量存储方式和地址偏移分配思路。虽然当前实现仍然是简化版本，但已经能够体现从符号表到内存映像的基本流程，也为后续实验八中的目标代码生成提供了基础。

本次实验说明，一个完整编译器不仅需要完成前端分析，还需要在后端阶段将变量、临时变量和中间代码映射到具体的运行时存储结构中。通过本次实验，我对编译器整体流程有了更加完整的认识。
